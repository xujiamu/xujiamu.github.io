性能和可维护性的选择

结论： 声明式代码的性能不优于命令式代码的性能

而对于Vue框架来说，它首先需要找到更新前后的差异，并只更新变化的地方，但最终完成更新的代码依然是上面那行

因此，如果将直接修改的性能定义为A，找出差异的性能定义为B，则：

  • 命令式代码的更新性能 = A
  • 声明式代码的更新性能 = B + A

既然性能命令式代码更好，那么选择声明式代码的原因在于，声明式代码的可维护性更强。（它不需要手动完成DOM的更新，创建，删除等操作）

基于以上框架设计者要做的是：在保持可维护性的同时，让性能损失最小化

  • 创建DOM时：
创建方式 性能
纯js操作DOM，

innerHTML： 在JQuery中或直接使用js编辑页面时，

  • 从表格可以看出创建DOM时，innerHTML和虚拟DOM性能接近，
  • 然而虚拟DOM的优势在于 更新DOM
    • innerHTML更新时需要：
      • 重新拼接HTML字符串，
      • 销毁所有旧DOM
      • 新建新DOM（由于是全部销毁重建， 所以模板越大，性能消耗越大）
    • 虚拟DOM：
      • 创建新的JS对象 + Diff (比较新旧虚拟DOM)
      • 只更新必要DOM

运行时和编译时

  • 设计框架有三种选择
    • 纯运行时框架
    • 运行时+编译时框架
    • 纯编译时框架

纯运行框架不支持使用类似HTML的结构，描述数据对象， 但如果引入编译的手段， 就可以实现编译+运行框架

那么如果直接根据需求将HTML结构，编译成命令式代码，就会变成纯编译框架

三种方式优缺点：

  • 纯编译框架，性能好，但由于内容必须编译后才能使用，所以缺乏灵活性
  • Vue3保留了 运行+编译的架构， 并且支持 运行时框架的使用方式

JavaScript 对象及 Proxy 的工作原理

JavaScript 中有两种对象，其中一种叫作常规对象（ordinary object），另一种叫作异质对象（exotic object）

在 JavaScript 中，函数其实也是对象。如何区分它是普通对象还是函数呢？通过内部方法（函数对象会部署内部方法[[Call]]，而普通对象则不会）

实际上，在 JavaScript 中，对象的实际语义是由对象的内部方法（internalmethod）指定的。所谓内部方法，指的是当我们对一个对象进行操作时在引擎内部调用的方法，这些方法对于 JavaScript 使用者来说是不可见的。

Proxy是一个异质对象, 创建代理对象时指定的拦截函数，实际上是用来自定义代理对象本身的内部方法和行为的，而不是用来指定被代理对象的内部方法和行为的

如果设置的属性不存在于对象上，那么会取得其原型，并调用原型的[[Set]] 方法，也就是 parent 的 [[Set]]。由于 parent 是代理对象，所以这就相当于执行了它的 set 拦截函数。前面提到过，parent的副作用函数已经被收集了，此处再触发，再加上child.bar本身的触发，所以一共执行两次

  • 解决方案：区分两次更新，屏蔽由原型引起的更新

    • 首先观察两次的拦截函数,可以发现 无论在什么情况下，receiver 都是 child，而 target 则是变化的

    • 因此只有当 receiver 是 target 的代理对象时才触发更新，这样就能够屏蔽由原型引起的更新了。

代理数组

数组是一个异质对象，数组对象的 [[DefineOwnProperty]] 内部方法与常规对象不同，其他内部方法的逻辑都与常规对象相同。因此，当实现对数组的代理时，用于代理普通对象的大部分代码可以继续使用

对数组操作的总结（下文将围绕这些操作实现代理）：

  • 读取操作
    • 通过索引访问数组元素值：arr[0]
    • 访问数组的长度：arr.length
    • 把数组作为对象，使用 for…in循环遍历。
    • 使用 for…of迭代遍历数组。
    • 数组的原型方法，如 concat/join/every/some/find/findIndex/includes等，以及其他所有不改变原数组的原型方法。
  • 设置操作
    • 通过索引修改数组元素值：arr[1] = 3
    • 修改数组长度：arr.length = 0
    • 数组的栈方法：push/pop/shift/unshift
    • 修改原数组的原型方法：splice/fill/sort等。

数组的索引与 length

当我们通过索引设置数组元素的值时，会执行数组对象所部署的内部方法 [[Set]]，由于内部方法 [[Set]]依赖于 [[DefineOwnProperty]]，所以产生了差异。

  • 具体体现在：

    • 当通过索引设置元素值时，可能会隐式地修改 length 的属性值。因此在触发响应时，也应该触发与 length 属性相关联的副作用函数重新执行
    • 反之亦然，修改数组的 length 属性也会隐式地影响数组元素 （但只有旧索引值大于或等于新 length 属性值， 元素才需要触发响应）

代理 Set 和 Map

由于集合类型数据的操作与普通对象存在很大的不同， 所以不能用原有方式代理它们，但基本思路不变

如何代理 Set 和 Map

问题一：代理对象调用size报错

  • 原因：Set.prototype.size 是一个访问器属性, 它首先会指定一个this 并将值赋给S, 由于通过代理对象 p来访问size属性，所以 this(S)就是代理对象 p。随后调用抽象方法RequireInternalSlot(S, [[SetData]])，检查S是否存在内部槽[[SetData]], 由于代理对象中不存在内部槽，所以抛出错误

  • 解决方案：修正this指向

问题二：代理对象删除报错

  • 原因：delete是一个方法。当访问 p.size 时，访问器属性的 getter函数会立即执行，此时我们可以通过修改 receiver来改变 getter函数的 this 的指向。而当访问 p.delete时，delete方法并没有执行，真正使其执行的语句是 p.delete(1)这句函数调用。因此，无论怎么修改 receiver，delete方法执行时的 this 都会指向代理对象 p，而不会指向原始 Set对象

  • 解决放案：把 delete方法与原始数据对象绑定

小结：将代码封装到createReactive函数中

避免污染原始数据

借助 Map类型数据的 set 和 get方法来讲解“避免污染原始数据”

首先实现get & set （与add, delete类似）

以set为例，来说明污染原始数据问题

  • 在副作用函数中，我们通过原始数据 m来读取数据值，然后又通过原始数据 m设置数据值，此时副作用函数重新执行。这个行为并不是我们所期望的。
    • 因为m作为原始数据，不应该具有响应式能力，否则就意味着用户既可以操作原始数据，又能够操作响应式数据，代码就会混乱
    • 导致这个问题的原因是 set实现中target.set(key, value)这行代码，将响应式数据直接设置到了原始数据上，这种行为就叫做数据污染
  • 解决方案：在设置前进行检查

1.目前为止，一直使用raw访问原始数据，但实际上它可能会与用户自定义的raw产生冲突，所以在更为严谨的实现中通过唯一标识（如Symbol）来代理它
2.Set类型的 add 方法、普通对象的写值操作，还有为数组添加元素的方法等，都需要做类似的处理
